# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

Tensotron is a PyTorch-faithful, **float32-only** GPU tensor + autograd library for .NET 8, built on **ILGPU**. See `README.md` for the full design rationale and current backend status; this file is the operational distillation.

## The law (non-negotiable)

**Tensotron mimics PyTorch in everything — naming, semantics, broadcasting, gradients.** If an op doesn't behave like PyTorch (down to behavior at kinks, ties, and special values), it's a bug, not a design choice. Converting PyTorch code to Tensotron should be near-mechanical. When in doubt about an op's name, signature, or edge behavior, match torch — don't invent.

## Build & test

```
tools/run-tests.ps1                          # preferred: torch-parity + fast smoke, at BelowNormal priority
tools/run-tests.ps1 -Filter "FullyQualifiedName~PoolTests"   # scope to a subset
tools/run-tests.ps1 -Showcase                # ONLY the slow Category=Showcase convergence demos (GPU)
dotnet test --filter "Category!=Showcase"    # plain-CLI equivalent of the default run
dotnet build Tensotron.sln --verbosity quiet # (use --verbosity quiet, never -q)
```

- `run-tests.ps1` runs at low process priority and first kills stray testhosts **scoped to this repo path** (safe on a shared machine). Prefer it over bare `dotnet test`.
- **A bare `dotnet test` with no filter runs everything, including the minutes-long showcase convergence tests.** The default `run-tests.ps1` excludes `Category=Showcase` but keeps the fast always-on `ShowcaseSmokeTests`.
- The default run uses `Auto`: CUDA if present, else the **managed/SIMD CPU backend** (not the slow ILGPU CPU accelerator) — so the full parity suite runs without a GPU. Showcase convergence demos gate on `Cuda.IsAvailable()` via `SkippableFact` and report **Skipped** on a CPU-only box.
- **No torch dependency to build or test.** torch is needed *only* to regenerate fixtures.

## Adding or changing an op (the core workflow)

Every op ships with a **golden-fixture parity test or it doesn't land** — asserting both **forward and backward** match torch within tolerance.

1. Implement the op in the relevant `src/Tensotron/TensorOps.*.cs` partial (Binary/Unary/Reduce/Movement/Matmul/Conv/Pool/Norm/Losses/Index/Select/Structural). Each op computes forward via a kernel, then (unless `NoGrad`) attaches a named `GradNode` whose closure deposits input gradients. Broadcast reduction in backward goes through `ReduceGradToShape`.
2. Add cases to `tools/fixtures/gen.py` and run `python tools/fixtures/gen.py` (the **only** place torch is needed) to emit JSON under `tests/Tensotron.Tests/Fixtures/`. The generator self-embeds its own source into each JSON for provenance.
3. **Fixtures must probe the hard parts, not just `randn` interior:** boundary inputs that hit the kink exactly (e.g. `x==0` for activations, `x==bound` for clamp), parameter sweeps that break shortcuts (e.g. LeakyReLU slope `<1`, `>1`, `<0`), and ties/special values (equal operands for min/max; exact-zero targets for KL). Edge cases use a deterministic `grad_output` (ones) so the recorded boundary gradient is unambiguous.
4. The C# test reads the committed JSON and asserts forward + backward parity — it never imports torch.

## Architecture (the big picture)

- **Storage/shape split.** A `Tensor` is `(Shape, TensorStorage, autograd metadata)`. Shape/strides stay host-side; data is `TensorStorage` (`DeviceStorage` = ILGPU `MemoryBuffer1D<float>`, or `HostStorage` = `float[]` on the managed backend). reshape/transpose/broadcast/expand are **zero-copy stride manipulation** (stride-0 marks a broadcast dim) — the matmul-backward transpose is a stride swap, not a copy. Materialize to host **only** on explicit `.ToArray()`/`.Item()`.
- **Autograd = explicit toposort.** Each op records a `GradNode { OpName, Inputs, Backward }`. `Backward()` topologically sorts the graph then runs node backward closures in reverse; a node is reached only after every consumer of its output, so its gradient is fully accumulated when its closure runs and accumulation is a plain `grad += …`. Interior grads are cleared at `Backward()` entry; **leaf grads accumulate across calls** (matching torch `retain_graph=True`) — call `ZeroGrad` between passes.
- **`TensorRuntime` is a process-wide singleton** (an abstract base; one backend per process, chosen once from `RequestedBackend` / `TENSOTRON_BACKEND`). `IlgpuRuntime` owns the single ILGPU `Context`/`Accelerator`, kernels compiled once via `LoadAutoGroupedStreamKernel` and **cached** — never recompiled per call. `Create()` resolves the backend: `Auto`→CUDA if a CUDA device is probed (`CudaPresent()`), else the managed/SIMD CPU backend. The op layer holds `Tensor.Buffer` as `TensorStorage` and the active runtime downcasts to the storage it owns. There is intentionally **no per-tensor device / `tensor.to(device)`**.
- **`CpuSimdRuntime` (`TENSOTRON_BACKEND=simd`)** — a hand-written, ILGPU-free managed/SIMD CPU backend (`CpuKernels` = the managed twin of `Kernels`, reusing the same `IOp` structs so XMath transcendentals match the GPU path). The *fast* CPU path for small-net inference/training: batch-1 control-net inference is ~645× faster than ILGPU's **scalar** CPUAccelerator (which carries full per-op dispatch overhead) and beats hand-scalar C# at batch≥8. SIMD is in `CpuKernels.MatMul2D` (`Vector<float>` dot + AXPY paths), with **opt-in row parallelism** (`TENSOTRON_CPU_THREADS=auto`/N, or `TensorRuntime.CpuMatMulThreads`; default 1 = serial, gated by a 1e6-FLOP floor `TENSOTRON_CPU_MINFLOPS`). Row-parallel matmul gives ~5–12× on big-batch GEMMs for a single trainer on idle cores, but *backfires* when cores are already saturated (many parallel agents) — so it defaults off; leave it off per-agent in multi-env RL, turn it on for one big-batch trainer. **`Auto` (default) falls back to this SIMD backend when no CUDA GPU is present**, not to the ILGPU CPU accelerator. The ILGPU `cpu` accelerator is a slow reference/verification device only (`TENSOTRON_BACKEND=cpu`) — `IlgpuRuntime` prints a loud `WarnIlgpuCpuIsSlow()` banner whenever the live accelerator is `AcceleratorType.CPU`. Regression gate: `tools/run-tests.ps1 -Simd`. See `docs/CPU_SIMD_BACKEND_{PLAN,PROGRESS}.md`.
- **Elementwise kernels are struct-generic.** ILGPU can't take a `Func`, so one arbitrary-rank strided kernel is parameterized by a `struct IOp { float Apply(...) }` that inlines at JIT. Global thread id → multi-dim index via strides; no rank ceiling.
- **Matmul** is a 2D GEMM core with rank/broadcast/autograd choreography in the portable layer: handles 1D@1D, 1D@2D, 2D@1D, 2D@2D, and N-D batched with broadcast batch dims; backward is `dA = dC@Bᵀ`, `dB = Aᵀ@dC` with batch-dim reduction. Three tiers behind one interface by size: **cuBLAS SGEMM** (`M,N,K ≥ 64`, CUDA — matches PyTorch FP32 at scale), a tiled shared-memory kernel, and a naive kernel for tiny products. Batched matmul above a work threshold uses **one `cublasSgemmStridedBatched`** when the batch reduces to constant element strides (incl. a fully-broadcast operand at stride 0) — a direct cuBLAS P/Invoke (`SgemmStridedBatched` delegate, resolved across DLL versions; ILGPU's CuBlas binds only single-matrix GEMM), gated by `UsesStridedBatchedGemm`/`EnableStridedBatchedGemm`; non-constant-stride batches fall back to a per-matrix SGEMM loop. Row-major↔column-major is the operand-swap identity (compute Cᵀ=Bᵀ·Aᵀ; cuBLAS-A=our B, cuBLAS-B=our A) — same mapping for the loop and the strided-batched call.

## Hard constraints

- **float32 only.** No other dtype — keeps the kernel surface small. Don't add dtype generality.
- **Single-threaded execution only (across ops).** `TensorRuntime` launch methods share accelerator state without internal serialization — do **not** call ops from multiple threads concurrently. The test suite disables xUnit parallelization for this reason (`AssemblyInfo.cs`). *Exception, within one op:* the SIMD backend's matmul can split output rows across worker threads (opt-in `CpuMatMulThreads`>1) — bit-identical to serial (disjoint row writes, same reduction order), so it doesn't relax this rule or parity.
- **Not bitwise-deterministic.** GPU reductions use atomics; results vary across runs. PPO-style training tolerates this. Don't assume reproducibility.
- **Buffer lifetime.** `Tensor` is `IDisposable` and owns its device buffer **except** zero-copy views (`Detach`, `Reshape`) which share the parent's buffer and never free it (`OwnsBuffer` tracks this; `Dispose()` is idempotent). Disposal is the deterministic opt-in for inference / no-grad loops; autograd intermediates are GC-reclaimed via ILGPU finalizers.
- **Async runtime, mostly de-overheaded (correctness still first).** Kernels launch on ILGPU's in-order default stream; the runtime `Synchronize()`s **only** at host pulls (`ToArray`/`Item`) plus a safety drain every `FlushEvery` launches (default 64, a settable knob) — NOT per launch. In place: cuBLAS GEMM, a size-bucketed caching allocator (fed by `Dispose`; per-step `DisposeGraph` recycle), shape/stride int buffers cached-by-content and uploaded once (`_intCache`), fused Adam/SGD kernels, chunked parallel reduction for few-output/large-extent gradients (conv bias), device-resident MaxPool argmax, device-resident `ClipGradNorm` (0 syncs). The dominant tiny-model cost is the **per-op host-side graph** — a fresh `Tensor`/`GradNode` per op + a full C# graph rebuild every step (**95% host-bound**, ~37 µs/op of pure graph construction). **Escape hatch:** `TensorRuntime.Capture(body)` records a fixed-shape step's device launches once and `CapturedGraph.Replay()` re-runs it with no host graph work. On CUDA the recorded launches (kernels and cuBLAS SGEMMs) are folded into one **native CUDA graph** — driver interop (`cuStreamBeginCapture`/`cuGraphLaunch` via `nvcuda`) on a dedicated owned stream, since ILGPU's default stream is the uncapturable NULL stream — replayed with a single `cuGraphLaunch`: **~5–10×/step vs eager** (vs ~2–2.7× for the host-side software replay it falls back to when a graph can't be built; toggle `TensorRuntime.EnableCudaGraph`). Capture taps each `Launch*` (only while capturing; an un-tapped op throws via the `AfterLaunch` `CallerMemberName` assertion), pins trace buffers against pool reuse (released on `CapturedGraph.Dispose`), and feeds new inputs via `Tensor.Upload`. Adam bias correction advances on the device (`adv=[t,invBc1,invBc2]`) so training stays exact across replays. Capture in **steady state**: persistent optimizer/state buffers must be allocated (one warmup step) before capture or their one-time zero-init replays and corrupts state — the same contract PyTorch CUDA graphs require. Eager execution stays on the default stream (host uploads/pulls ordered with kernels); `_stream` is swapped to the graph stream only during capture. Limits: a scheduled-LR scalar baked into a kernel at capture is frozen; no data-dependent index ops (maxpool argmax, gather). Cross-op kernel fusion is unimplemented. See `docs/PERFORMANCE_VS_PYTORCH.md`.

## Layout

- `src/Tensotron/` — `Shape`, `Tensor`, `TensorOps.*` (op surface), `Kernels`, `TensorRuntime`, `Ops*` (struct-generic op types), plus `Module`/`Conv`/`Pool`/`Norm`/`Optimizers`/`LrSchedulers`/`Init`/`DataLoader`/`Serialization`. Packable as NuGet package `Tensotron` (uses `PACKAGE.md` as the packed readme, not `README.md`) — **not yet published to nuget.org**; consumers use a `ProjectReference` or a locally-packed `.nupkg`.
- `tests/Tensotron.Tests/` — torch-parity tests + `Fixtures.cs` loader; `Fixtures/*.json` committed. "Tensotron works."
- `showcase/Tensotron.Showcase/` — end-to-end tasks (continuous-PPO pole-cart, MNIST CNN) asserting real learning, emitting SVG replays. "Tensotron can be used for these things."
- `examples/Tensotron.Examples/` — runnable console app (`dotnet run --project examples/Tensotron.Examples [xor|spiral|regression]`) with minimal from-scratch training loops for newcomers; spiral/regression write an SVG. Not a test project. `Plot.cs` is a local SVG helper, not part of the library.
- `tools/fixtures/gen.py` — torch fixture generator (the only torch dependency).
